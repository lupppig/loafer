# Pipeline Configuration
Last updated: March 2025  ·  5 min read  ·  [Edit on GitHub ↗](https://github.com/lupppig/loafer/edit/main/web/src/content/docs/pipelines.mdx)

The Pipeline configuration file is the heart of Loafer. By default, Loafer looks for a `pipeline.yaml` in your execution directory.

## Config file overview

A pipeline consists of at least a `source`, `target`, and `transform` node. 

| Key | Type | Required | Default | Description |
|---|---|---|---|---|
| `version` | string | Yes | - | The version of the pipeline syntax (currently 1.0) |
| `mode` | string | Yes | - | Either `etl` (extract, transform, load) or `elt` (extract, load, transform) |
| `source` | object | Yes | - | The source connector configuration |
| `target` | object | Yes | - | The target connector configuration |
| `transform` | object | No | null | The logic for transforming data |
| `schedule` | object | No | null | Defines periodic execution logic |

## Mode: ETL vs ELT

Loafer supports both ETL and ELT modalities. Your choice will drastically impact how processing occurs.

```text
ETL Mode
[Source] -----> (Local Node Transform) -----> [Target]
                 • Row by row stream
                 • AI/Python/SQL Local Evaluation

ELT Mode
[Source] -----------------------------------> [Target]
                 • Raw data streamed directly
                 • SQL executed on Target Node
```

In **ELT mode**, transformations using Custom Python or AI are disabled. Compute relies entirely on the destination database engine. 

## Full config reference

### `source`

Defines where to extract data from.

| Key | Type | Required | Default | Description |
|---|---|---|---|---|
| `type` | string | No | auto-detected | The connector identifier (`postgres`, `csv`, `mongodb`, etc.). Auto-detected from `url` scheme or file extension. |
| `url` | string | Yes* | - | The connection string. Usually required. |
| `query` | string | No | null | The SQL/NoSQL query to execute for extraction. |

> **Auto-detection:** When `type` is omitted, Loafer infers the connector from the `url` scheme (e.g. `postgresql://` → `postgres`, `mongodb://` → `mongo`) or the file `path` extension (e.g. `.csv` → `csv`, `.xlsx` → `excel`). You can always set `type` explicitly to override.

```yaml
# Type auto-detected as postgres from URL scheme
source:
  url: ${DATABASE_URL}
  query: "SELECT * FROM orders WHERE status = 'paid'"
```

### `target`

Defines where to load data to.

| Key | Type | Required | Default | Description |
|---|---|---|---|---|
| `type` | string | No | auto-detected | The connector identifier (`postgres`, `json`, `csv`, etc.). Auto-detected from `url` scheme or file extension. |
| `url` | string | Yes* | - | The connection string or filepath. |
| `mode` | string | No | append | Target write mode (`append`, `overwrite`). |

```yaml
# Type auto-detected as json from .json file extension
target:
  path: ./output/data.json
```

## Environment variable interpolation

You can safely interpolate environment variables anywhere in the pipeline using the `${VAR_NAME}` syntax.

```yaml
source:
  url: ${PRODUCTION_DB_URL}
```

If you specify an environment variable that is not loaded via the local environment or a `.env` file, the `loafer run` command will fail early during configuration validation.
